B- 树和 B+ 树

发表于 2021-09-03 分类于 Mysql ， 索引 阅读次数：
本文字数： 1.4k 阅读时长 ≈ 1 分钟

聚簇索引和非聚簇索引

B- 树（又称为 B 树）

B- 树是一种平衡的多路查找树，主要用作文件的索引，因此它的查找设计外存（主要是硬盘）的存取。

一颗 m 阶的 B- 树，或为空树，或为满足下列特性的 m 叉树：

  1. 树中的每个节点至多有 m 颗子树；
  2. 若根节点不是叶子节点，则至少有两颗子树；
  3. 除根节点之外的所有非终端结点至少有 m/2(上去整) 颗子树；
  4. 所有非终端节点中包含下列数据信息 ($n$​，$A_0$​，$K_1$​，$A_1$​，$K_2$​，$A_2$​，$\cdots$​，$K_n$​，$A_n$​)；K 为关键字，A 为指向子树根节点的指针，且 $A_{i-1}$​ 所指子树中的所有结点的关键字都均小于 $K_i$​，$A_n$​ 所指子树中的所有结点均大于 $K_n$​，n($\lceil{m/2}\rceil-1 \leq n \leq m-1$​​ )，为关键字的个数（n+1 为子树个数）；
  5. 所有叶子节点出现在同一层次上，并且不带信息（可以看作外部节点或者查找失败的节点，实际上这些节点并不存在，指向这些节点的指针为空）；

B- 树查找分析

主要包含两种操作：

  1. 在 B- 树中查找结点；
    由于 B- 树通常存储在磁盘上，则这一查找操作是在磁盘上进行的。
  2. 在结点中查找关键字；
    找到结点后，将结点中的信息读入内存，再利用顺序还是折半查找等于 K 的关键字。

分析：因为在磁盘上的一次查找比内存中的费时多得多，所以待查关键字在 B- 树上的层次是决定 B- 查找效率的首要因素。

B+ 树

B+ 树应文件系统所需而出的一种 B- 树的变型树。一颗 m 阶的 B+ 树和 m 阶的 B- 树差异在于：

  1. 有 n 颗子树的结点中含有 n 个关键字；
  2. 所有叶子节点中包含了全部关键字的信息，及指向这些关键字的指针，且叶子结点本身依关键字的大小自小而大顺序链接；
  3. 所有非终端节点可以看成索引部分，节点中仅含有其子树（根节点）中的最大或最小关键字

通常 B+ 树上有两个头指针，一个指向根节点，另一个指向关键字最小的叶子节点。因此对 B+ 树查找有两种：一种是从最小关键字顺序查找；另一种是从根节点随机查找。

B+ 树的随机查找、插入、删除过程基本和 B- 树类似，只是在查找时，若非终端结点上的关键字等于给定值，并不终止，而是继续向下到叶子结点，因此 B+ 树的查找不论成功与否，都是走了一条从根节点到叶子节点的路径。

MySQL 索引理解

索引：可以提高数据检索的效率，降低数据库的 IO 成本。

聚簇索引：将数据存储与索引放到了一块，找到索引就找到了数据。

非聚簇索引：数据存储和索引分开放，索引结构的叶子节点指向了数据的对应行，myisam通过 key_buffer 把索引先缓存到内存中，当需要访问数据时（通过索引访问数据），在内存中直接搜索索引，然后通过索引找到磁盘相应数据，这也就是为什么索引不在 key buffer 命中时，速度慢的原因（磁盘 IO）。

索引和数据在一起就是聚簇索引，不在一起就叫非聚簇索引

5.7以后的mysql默认引擎都是InnoDB

在索引的分类中，我们可以按照索引的键是否为主键来分为“主索引”和“辅助索引”

因此：主索引只能有一个，辅助索引可以有很多个，只有主键索引为主索引，其他都是辅助索引；

InnoDB中主键索引为聚簇索引，辅助索引是非聚簇索引：这句话的理解：

主键索引中通过B+树找到的叶子节点的数据包含了所有的数据；

辅助索引的叶子节点的数据仅仅包含了辅助节点的数据，不包含所有的数据，如果需要索引外的内容，则需要回表；
